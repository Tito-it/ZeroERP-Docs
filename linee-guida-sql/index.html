    1. Schemi Principali¶

    Organizzati per funzionalità e responsabilità:¶

    Schema Descrizione
    anagrafiche Contiene tutte le tabelle legate all'anagrafica soggetti (persone fisiche e giuridiche).
    • Esempi: soggetti, indirizzi, contatti
    • Usa identificatori UUID e naming coerente
    config Contiene tabelle di configurazione parametrica e lookup.
    • Esempi: stati, nazioni, contatti_ruoli
    • Normalizzazione forte, spesso referenziate da altri schemi
    script Logica applicativa (viste e funzioni) legate ai processi ERP. Favorisce riuso, mantenibilità e isolamento della logica.
    • Esempi: vw_clienti_attivi, fn_calcolo_fattura()
    • Include anche: fn_get_stradario_by_indirizzo(), etc.
    dbops Strumenti DB per DBA (vista/funzioni tecniche, metadata, monitoring).
    • Esempi: vw_function_list, fn_foreign_keys_info(text)
    • Nessuna logica applicativa, solo strumenti per DBA e manutentori
    storico Tabelle di audit/storico con versioning temporale dei dati. Tutte le tabelle che richiedono "storicizzazione" saranno replicate qui
    • Esempi: soggetti_storico, contratti_storico
    • Usato per audit, confronto tra versioni, report du dati storici
    public Mostra solo tabelle pubbliche (non pubbliche), ad esempio version_info
    • Esempi: version_info, ecc...
    • Solo tabelle o funzioni non pubbliche

    2. Convenzioni di Naming¶

    Uniformità e chiarezza nel codice DDL:

    Oggetto Convenzione Esempio
    Schema minuscolo, plurale anagrafiche, config, dbops
    Tabella nome plurale, snake_case soggetti, gestori_utilities_ruoli
    Colonna PK id id UUID, id SERIAL
    Sequenza <table>_id_seq storico.comuni_id_seq
    FK <referenced_table>_id soggetto_id, utility_id
    Indice ix_<table>_<column> ix_contatti_soggetto
    Vista prefisso vw_ vw_clienti_attivi
    Funzione prefisso fn_ fn_calcolo_fattura()
    Trigger Func. prefisso trgfn_ trgfn_contatti_set_default()
    Trigger prefisso trg_, <table>_<event> trg_contatti_insert_update
    Constraint PK pk_<table> pk_soggetti_ruoli
    Constraint FK fk_<base>_<ref> fk_contatti_soggetti
    Constraint UQ uq_<table>_<columns> uq_contatti_default

    ℹ️ Le convenzioni sono tratte dalla struttura condivisa in Zero ERP

    3. Gestione delle Chiavi Primarie¶

    1. UUID per entità master (anagrafiche.soggetti):

    id UUID PRIMARY KEY
  DEFAULT gen_random_uuid()
    2. Serial/Identity per tabelle tecniche e di relazione:

    id SERIAL PRIMARY KEY
-- o
id INTEGER GENERATED BY DEFAULT AS IDENTITY
    3. Sequenze esplicite quando serve:

    • Creazione: CREATE SEQUENCE schema.table_id_seq START 1;
    • Colonna: defaultValueComputed: "nextval('schema.table_id_seq'::regclass)"
    • Oppure autoIncrement: true per delegare a Liquibase/DB

    4. Eccezioni e Alternative¶

    • UUID vs SERIAL: usare UUID quando serve unicità globale o sharding; SERIAL su tabelle di log e storicizzazione.
    • Foreign Key con composti: preferire surrogate key (id) e colonne FK (<table>_id) piuttosto che chiavi composte.
    • Campi tecnici: evitare prefissi in dbops; usare solo vw_... e fn_... per uniformità

    5. Trigger e Funzioni¶

    • Trigger Functions (PL/pgSQL, schema.trgfn_<table>_<descr>):

    • Esempio: CREATE OR REPLACE FUNCTION anagrafiche.trgfn_contatti_set_default()

    • Trigger (DDL):

    CREATE TRIGGER trg_contatti_set_default
  BEFORE INSERT OR UPDATE
  ON anagrafiche.contatti
  FOR EACH ROW
  EXECUTE FUNCTION anagrafiche.trgfn_contatti_set_default();
    * Nomi:

    • Funzioni: trgfn_<table>_<action>
    • Trigger: trg_<table>_<action>

    • Best practice:

    • Mettere funzioni e trigger nello stesso schema della tabella (se dedicate), o in script/dbops per utilizzabilità multipla.

    • Documentare sempre parametri e comportamento

    • Funzioni di Cleanup (PL/pgSQL, schema.trgfn_<table>_ins_upd_clean_fields):

    • Usate per pulire, normalizzare o impostare valori di default sui campi mancanti prima di INSERT o UPDATE.
    • Convenzione di naming:
      • Funzione: trgfn_<table>_ins_upd_clean_fields
      • Trigger: trg_<table>_ins_upd_clean_fields
    • Esempio: 
      CREATE OR REPLACE FUNCTION anagrafiche.trgfn_soggetti_ins_upd_clean_fields()
  RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  IF NEW.stato_id IS NULL THEN
    NEW.stato_id := script.fn_get_default_by_param(
      'config', 'stati', 'codice_iso3', 'state_code_iso3', TG_NAME
    )::integer;
  END IF;
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_soggetti_ins_upd_clean_fields
  BEFORE INSERT OR UPDATE ON anagrafiche.soggetti
  FOR EACH ROW
  EXECUTE FUNCTION anagrafiche.trgfn_soggetti_ins_upd_clean_fields();

    6. Viste (Views)¶

    • Applicative: vw_<descrittivo> in schema script, es. vw_clienti_attivi
    • Tecniche: in schema dbops, senza prefisso extra, es. vw_function_list

    7. Colonne di Audit e Log¶

    Standardizzare i campi di auditing:

    date_insert    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP WITH TIME ZONE
user_insert    TEXT      NOT NULL DEFAULT CURRENT_USER
data_update    TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP WITH TIME ZONE
user_update    TEXT      NOT NULL DEFAULT CURRENT_USER

    I trigger o constraint di sistema possono aggiornare date_update/user_update su UPDATE manuale o con trigger dedicato.

    8. Indici¶

    • Naming: ix_<table>_<column>
    • Tipologia: B-Tree per default; GIN/GIST per JSONB/Geometrie.
    • Parziali: CREATE UNIQUE INDEX ix_contatti_default ON contatti(soggetto_id) WHERE is_default; per default logic

    9. Automazione Indici FK¶

    All’interno della cartella 90_end/ delle migrazioni Liquibase, inserire sempre per ultimi uno script YAML con runAlways o runAlways: false per richiamare la procedura che crea/rimuove indici sulle foreign key:

    - changeSet:
    id: 90-end-apply-fk-policy
    author: mctit
    runAlways: true
    sql:
      splitStatements: false
      sql: |
        CALL dbops.apply_fk_index_policy();

    • La tabella dbops.fk_index_policy deve essere mantenuta aggiornata:

    • individuare nuove FK senza indice con dbops.vw_fk_senza_indice

    • popolare policy con SELECT dbops.fn_populate_fk_index_policy();
    • La policy consente di generare o eliminare indici non necessari.

    10. Gestione Indici Duplicati¶

    Utilizzare la view dbops.vw_duplicate_indexes per visualizzare gli indici che condividono lo stesso insieme di colonne e razionalizzare la struttura:

    SELECT *
  FROM dbops.vw_duplicate_indexes;

    11. Transaction ID Centralizzato¶

    Per garantire coerenza transazionale fra tutte le tabelle che registrano modifiche, ogni entità deve avere la colonna tx_id UUID che accetta anche valori nulli:

    • Trigger (opzione DB): applicare trg_assign_tx_id (su ogni tabella) che utilizza la function dbops.trgfn_tx_id_managed() per generare o recuperare il tx_id di sessione.
    • Function (opzione backend): invocare dbops.fn_get_or_create_session_tx_id() prima delle operazioni e passare il tx_id in tutti gli INSERT/UPDATE.

    Per maggiori dettagli e implementazione, consulta la guida (è fondamentale la consultazione): Guida Tecnica – Implementazione di un Unico Transaction ID
